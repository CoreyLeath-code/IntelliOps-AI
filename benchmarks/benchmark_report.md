# IntelliOps-AI Research Benchmark Report

## Executive summary

This report benchmarks the actual lazy-trained PyTorch prediction path in `services/ml-model-pytorch/model.py`. Results were generated by GitHub Actions on 2026-07-18 with a fixed seed and committed as machine-readable evidence.

## Experimental protocol

- Fixed seed: `20260718`
- Runtime: CPython 3.11.15, PyTorch 2.6.0+cu124, CPU mode
- Runner: GitHub-hosted Linux runner, 4 reported CPUs
- Workload: 2,000 timed single-sample predictions after 100 warm-up predictions
- Dataset: scikit-learn Iris, binary target `target == 0`
- Cold start: first `predict` call, including the service's 100-epoch lazy training
- Percentiles: linearly interpolated ordered observations
- Python memory: `tracemalloc`
- Process memory: `resource.getrusage(...).ru_maxrss`

Reproduce with:

```bash
pip install -r requirements.txt
python benchmarks/run_benchmark.py --output benchmarks/latest.json
```

## Interpretation

Warm model execution is fast in this isolated, single-process CPU benchmark, while lazy initialization dominates first-request latency. The most valuable next optimization is moving training out of request handling and loading a versioned model artifact during controlled startup.

The quality score is intentionally a **training-corpus sanity check**: the service trains on all 150 Iris rows and this harness evaluates those same rows. It verifies deterministic pipeline behavior but is not held-out accuracy, a generalization estimate, or production model evidence.

## Regression policy

CI validates the evidence schema, requires positive latency/throughput measurements, and fails if the sanity-check F1 drops below 0.90. On comparable runners, investigate median or P95 regressions above 15%. Baseline updates must record the environment and explain the change.

## Limitations and next experiments

- HTTP serialization, Go gateway, containers, network, and concurrent load are excluded.
- Maximum RSS includes Python, PyTorch, and scikit-learn imports.
- GPU utilization and GPU memory are not applicable to this CPU run.
- A held-out, stratified evaluation must replace the training-corpus score before making model-quality claims.
- Scalability at 1/10/100 concurrent clients and end-to-end service latency remain separate planned experiments.
